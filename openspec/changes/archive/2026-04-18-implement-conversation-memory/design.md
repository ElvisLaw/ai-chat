## Context

当前 AI Chat 使用 `InMemoryConversationStore`，会话数据存在内存中，服务重启后丢失。且长对话会无限累积消息，可能导致：
- Token 超出 LLM 上下文限制
- 内存占用持续增长
- 检索效率下降

需要实现对话记忆功能，支持会话持久化和智能摘要。

## Goals / Non-Goals

**Goals:**
- 支持对话历史跨服务重启持久化
- 自动管理对话长度，防止超出 Token 限制
- 长对话自动生成摘要，保留关键信息
- 最小化实现复杂度，不引入额外数据库

**Non-Goals:**
- 不实现多租户隔离（单用户场景）
- 不实现实时协作或分布式存储
- 不实现向量检索或语义搜索（未来扩展）

## Decisions

### Decision 1: 持久化存储格式

**选择**: JSON 文件存储在 `~/.ai-chat/conversations/`

**理由**:
- 实现简单，无需额外部署
- 便于调试和人类可读
- 后续可平滑迁移到 SQLite

### Decision 2: 摘要触发时机

**选择**: 当消息数 > 15 且 Token 数 > 3000 且尚未摘要过时触发摘要

**理由**:
- 避免频繁摘要影响性能
- 3000 Token 留出足够空间给新消息
- 保留最新 5 条消息维持近期上下文
- 标记已摘要防止重复摘要

### Decision 3: Token 计数方式

**选择**: 简单字符数 / 4 估算（不引入 tiktoken）

**理由**:
- 无需额外依赖
- 对于触发阈值足够准确
- 后续可升级为精确计数

### Decision 4: 摘要生成

**选择**: 调用现有 LLM 生成摘要

**理由**:
- 复用现有 `create_llm_client` 接口
- 摘要质量优于规则提取
- 统一错误处理

### Decision 5: 缓冲与摘要关系

**选择**: 缓冲（20条）和摘要（15条+3000 tokens）是互补策略

**理由**:
- 缓冲处理快速到达上限的场景
- 摘要处理缓慢累积到上下文限制的场景
- 摘要后保留 5 条最新消息 + 摘要

### Decision 6: 持久化时机

**选择**: 每条消息后立即持久化（内存优先，失败不阻塞）

**理由**:
- 实现简单，一致性保证
- 对话文件较小，性能开销可接受
- 失败时下次重试，不阻塞对话

### Decision 7: 摘要长度控制

**选择**: 摘要不超过 500 字

**理由**:
- 足够包含核心信息
- 避免摘要本身占用过多上下文

## Responsibilities

| 组件 | 职责 |
|------|------|
| `FileConversationStore` | JSON 文件的读写、列表查询、删除 |
| `ConversationService` | 消息缓冲、Token 估算、摘要触发判断 |
| `MemorySummarizer` | 调用 LLM 生成摘要 |

## Risks / Trade-offs

| Risk | Impact | Mitigation |
|------|--------|------------|
| 摘要丢失关键信息 | 用户体验下降 | 保留摘要 + 最近 5 条消息 |
| 持久化失败 | 数据丢失 | 内存优先，持久化失败不阻塞，下次重试 |
| Token 估算不准确 | 超出上下文限制 | 保守估计，预留 buffer |
| 重复摘要 | 上下文冗余 | 标记已摘要状态，增量摘要 |
